Grundlage 1 · Skip-Link .bs-skip live auf dieser Seite data-theme="dark"

Grundlagen · Barrierefreiheit

Barrierefrei ist Pflicht — und guter Stil

Seit dem 28. Juni 2025 gilt das Barrierefreiheitsstärkungsgesetz (BFSG): Wer online verkauft, bucht oder Termine vergibt, muss barrierefrei sein. Diese Seite bündelt die Muster, die im Baukasten bereits stecken — zum Nachschlagen und Kopieren. Erster Baustein: der Skip-Link, hier live eingebaut.

Probier es auf dieser Seite aus

  1. Seite neu laden, dann einmal Tab drücken.
  2. Oben links erscheint „Zum Inhalt springen“ — der erste fokussierbare Punkt der Seite.
  3. Enter springt direkt zu <main id="inhalt"> und überspringt Kopfbereich samt Navigation.
<!-- Erstes Element im <body> — vor Nav und allem anderen: -->
<a class="bs-skip" href="#inhalt">Zum Inhalt springen</a>

<!-- … Navigation, Header … -->

<!-- tabindex="-1": das Ziel kann den Fokus wirklich übernehmen -->
<main id="inhalt" tabindex="-1"> … </main>

<style>
  /* Unsichtbar geparkt — display:none wäre falsch,
     denn dann wäre der Link nicht mehr fokussierbar. */
  .bs-skip { position: fixed; top: 16px; left: 16px; z-index: 1000;
    padding: 13px 22px; border-radius: 999px;
    font: 700 15px/1 var(--font-body); color: #fff; background: var(--brand);
    transform: translateY(calc(-100% - 24px)); transition: transform .18s ease; }
  .bs-skip:focus-visible { transform: translateY(0);
    outline: 2px solid #fff; outline-offset: 3px; }
  /* Sprungziel = Container, kein Bedienelement → Ausnahme erlaubt */
  main#inhalt:focus-visible { outline: none; }
</style>

Grundlage 2 · Fokus sichtbar machen :focus-visible

Fokus sichtbar machen

Wer mit der Tastatur navigiert, sieht nur am Fokus-Ring, wo er gerade steht. base.css bringt dafür einen globalen :focus-visible-Stil mit — Maus-Klicks bleiben davon unberührt.

Demo: Mit Tab durch die vier Elemente wandern — jedes zeigt einen deutlichen Fokus-Ring. (Demo-Füller: NICHT mitkopieren.)

Ein Textlink
⚠️

outline: none ist verboten, solange kein gleichwertiger Ersatz folgt: Ohne sichtbaren Fokus verlieren Tastatur-Nutzer sofort die Orientierung — Verstoß gegen WCAG 2.4.7 („Focus Visible“).

/* Fundament (steht so in base.css) — EIN Ring für alles Fokussierbare.
   :focus-visible statt :focus → erscheint bei Tastatur, nicht bei Maus. */
:focus-visible { outline: 2px solid var(--brand-300); outline-offset: 3px; }

/* Eigener Fokus-Stil? Immer ERSETZEN, nie entfernen: */
.bs-input:focus-visible {
  outline: none;                /* nur ok, weil direkt Ersatz folgt: */
  border-color: var(--brand);
  box-shadow: 0 0 0 3px rgba(var(--brand-rgb),.18);
}

/* Custom-Checkbox: Input ist unsichtbar, aber fokussierbar —
   der Ring wandert auf die sichtbare Box daneben. */
.democheck input:focus-visible + .box {
  outline: 2px solid var(--brand-300); outline-offset: 3px;
}

Grundlage 3 · ARIA-Spickzettel 5 Muster, fertig im Baukasten

Spickzettel

ARIA-Muster im Baukasten

Erste Regel von ARIA: kein ARIA, wenn natives HTML reicht. So sind die fünf häufigsten Muster hier gelöst — mit dem jeweiligen Kern-Attribut zum Kopieren.

▾Akkordeon

Natives <details>/<summary> — Tastaturbedienung, Semantik und Auf/Zu-Zustand liefert der Browser, null eigenes ARIA nötig.

Im Baukasten: faq.html · interaktiv.html

<details class="bs-faq-item">
  <summary>Was kostet der Tarif?</summary>
  <p>Antwort …</p>
</details>

⊟Tabs

Rollen-Trio tablist / tab / tabpanel; aria-selected markiert den aktiven Reiter, aria-controls verweist aufs Panel.

Im Baukasten: interaktiv.html · medien.html

<div role="tablist" aria-label="Ansichten">
  <button role="tab" id="tab-1"
    aria-selected="true" aria-controls="panel-1">…</button>
</div>
<div role="tabpanel" id="panel-1"
  aria-labelledby="tab-1">…</div>

▣Dialog / Modal

Natives <dialog> mit showModal(): Fokus-Falle, Esc zum Schließen und Hintergrund-Sperre gibt es geschenkt.

Im Baukasten: interaktiv.html · medien.html · leadgen.html

<dialog id="dlg" aria-labelledby="dlg-titel">
  <h2 id="dlg-titel">Demo buchen</h2> …
</dialog>
<script>
  document.getElementById("dlg").showModal();
  // Fokus-Falle + Esc kommen vom Browser
</script>

⇄Toggle-Button

Ein echter <button> mit aria-pressed — Screenreader lesen „gedrückt / nicht gedrückt“ vor. Der Wert wird bei jedem Klick umgeschaltet.

Im Baukasten: leadgen.html · social-proof.html

<button type="button" aria-pressed="false">
  Jährlich zahlen
</button>
// im Klick-Handler:
btn.setAttribute("aria-pressed",
  String(btn.getAttribute("aria-pressed") !== "true"));

📣Live-Region

Statusmeldungen (z. B. „Formular gesendet“) werden nur vorgelesen, wenn der Container vor der Änderung schon als Live-Region im DOM steht.

Im Baukasten: leadgen.html · danke.html · utility.html

<!-- role="status" wirkt wie aria-live="polite" -->
<p class="bs-success-box" role="status">
  Danke! Deine Anfrage ist angekommen.
</p>

✓Merksatz

Natives HTML zuerst. <button> statt klickbarem <div>, <details> statt Akkordeon-Skript, <dialog> statt Overlay-Eigenbau. ARIA repariert kein kaputtes Markup — es ergänzt nur, was HTML nicht ausdrücken kann.

Referenz: „ARIA Authoring Practices Guide“ (APG) des W3C

Grundlage 4 · Reduced Motion @media (prefers-reduced-motion: reduce) data-theme="dark"

Bewegung

Reduced Motion respektieren

Für Menschen mit vestibulären Störungen sind Parallax, Auto-Marquees und große Einflug-Animationen nicht lästig, sondern körperlich unangenehm. Die Systemeinstellung „Bewegung reduzieren“ meldet das per Media Query — der Baukasten reagiert global darauf.

Regel: Endzustand statt Animation

Nichts ausblenden, nichts weglassen — jedes animierte Element zeigt sofort seinen fertigen Zustand. Reveals stehen auf opacity: 1, Zähler zeigen direkt den Zielwert, Marquees stehen still.

  • Gilt automatisch für alle Bausteine — der Block steht einmal in base.css.
  • Auch das JS prüft die Einstellung: base.js überspringt Reveal-Observer und Zähler-Animation.
  • Zum Testen: macOS „Bewegung reduzieren“ bzw. in den DevTools „Emulate CSS prefers-reduced-motion“.
/* Globales Muster (steht so in base.css) —
   einmal einbinden, wirkt auf alle Bausteine: */
@media (prefers-reduced-motion: reduce) {
  html { scroll-behavior: auto; }
  *, *::before, *::after {
    animation-duration: .001ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: .001ms !important;
  }
  /* Endzustand statt Animation: Reveals sofort sichtbar */
  html.js [data-reveal] { opacity: 1; transform: none; }
}

/* Und im JS (base.js): Einstellung einmal lesen … */
const reduceMotion =
  matchMedia("(prefers-reduced-motion: reduce)").matches;

/* … und z. B. beim Zähler den Zielwert direkt setzen: */
if (reduceMotion) {
  el.textContent = zielwert.toLocaleString("de-DE");
} // sonst: requestAnimationFrame-Animation

Grundlage 5 · Kontrast 4,5:1 Fließtext 3:1 groß & UI

Farbe & Lesbarkeit

Kontrast: die zwei Faustregeln

Hübsch-dezent ist oft schlicht unlesbar. Vier Kacheln aus den Tokens des Baukastens — zwei bestehen, zwei fallen durch.

Klare Sache --fg auf --card ≈ 18,1:1 · AA + AAA bestanden
Klare Sache Weiß auf --brand (CTA-Button) ≈ 5,8:1 · AA bestanden
Wird knapp --fg-muted auf --bg ≈ 3,4:1 · fällt bei Fließtext durch — nur für großen Text ok
Unlesbar Marke mit 35 % Deckkraft ≈ 1,7:1 · fällt überall durch

Werte gemessen mit der Standard-Palette auf heller Fläche — nach jedem Farbschema-Wechsel neu prüfen.

4,5:1

Mindestkontrast für Fließtext und alles unter 24 px (bzw. 18,7 px fett) — WCAG-Stufe AA.

3:1

Reicht für großen Text (ab 24 px / 18,7 px fett) sowie für UI-Elemente wie Icons, Rahmen von Eingabefeldern und Fokus-Ringe.

Prüfen

Werkzeuge: WebAIM Contrast Checker, Chrome DevTools (Farb-Picker zeigt das Kontrastverhältnis), Firefox Barrierefreiheit-Inspektor, WAVE.

Grundlage 6 · Formular-A11y label/for aria-describedby aria-invalid data-theme="dark"

Formulare

Fehler, die man hören kann

Drei Attribute machen aus dem Formular-Fundament ein barrierefreies Formular: label/for benennt das Feld, aria-describedby verknüpft die Fehlermeldung, aria-invalid markiert den Zustand.

Live-Demo: So klingt und aussieht der Fehlerfall. (Der Fehlerzustand ist hier zur Ansicht fest gesetzt — Demo-Füller, NICHT mitkopieren.)

Bitte gib eine vollständige E-Mail-Adresse an, z. B. max@beispiel.de.

✓ Nach dem Absenden landet die Bestätigung in so einer Live-Region.

<!-- novalidate: eigene Fehlertexte statt Browser-Blasen -->
<form novalidate>
  <div class="bs-field">

    <!-- 1) label ↔ for="id": Screenreader-Name + große Klickfläche -->
    <label for="anfrage-email">E-Mail *</label>

    <!-- 2) aria-describedby verknüpft die Fehlermeldung: sie wird
         beim Fokussieren des Feldes mit vorgelesen.
         3) aria-invalid erst per JS auf "true" setzen, wenn die
         Prüfung fehlschlägt — sonst das Attribut entfernen. -->
    <input class="bs-input" type="email"
           id="anfrage-email" name="email"
           required autocomplete="email"
           aria-describedby="anfrage-email-err"
           aria-invalid="true" />

    <!-- Fehlertext: konkret sagen, WAS zu tun ist -->
    <p class="bs-error" id="anfrage-email-err">
      Bitte gib eine vollständige E-Mail-Adresse an.
    </p>
  </div>

  <!-- Erfolgsmeldung als Live-Region: steht LEER schon im DOM,
       role="status" liest den später eingesetzten Text vor -->
  <p class="bs-success-box" role="status" hidden></p>
</form>
