¿Codex dentro de Claude Code, o los dos como compañeros de equipo?

Bloome·5 de julio de 2026·5 min read

Hace unos días OpenAI publicó codex-plugin-cc: una forma de usar Codex desde dentro de Claude Code —para revisar código o delegarle una tarea— y escaló rápido en las listas de tendencias de GitHub. Vale la pena detenerse a pensar por qué, porque lo interesante no es el plugin en sí. Es lo que el plugin reconoce.

Si incluso el equipo detrás de uno de los principales agentes de programación está publicando una forma limpia de incorporar a un agente rival, la era del «un solo modelo para gobernarlos a todos» ya terminó en la práctica. El trabajo real necesita más de un especialista. Quieres Claude Code para una cosa y Codex para otra, y los quieres trabajando en el mismo problema — no tú alternando entre dos ventanas, copiando contexto de un lado al otro a mano.

Ese es un buen instinto. La única pregunta que queda es qué forma le das.

Qué hace el plugin en realidad

codex-plugin-cc es un plugin para Claude Code. Una vez instalado, Claude Code recibe un conjunto de slash commands respaldados por Codex — una revisión de código, una revisión de diseño adversarial, una delegación de «rescate» cuando una tarea se está torciendo, más comandos de trabajo en segundo plano para comprobar y recoger resultados. Por debajo, maneja tu Codex CLI local en la misma máquina.

Es genuinamente útil, y es honesto sobre lo que es: Codex, accesible desde dentro de Claude Code, para un desarrollador en su terminal.

Pero fíjate en la forma. El plugin anida un agente dentro del arnés del otro. Codex se convierte en una herramienta que Claude Code invoca. El esquema es:

  • una máquina — los dos agentes corren en local, para la persona en esa terminal;
  • un operador — tú eres el único humano en el bucle, y el único que puede ver lo que pasó;
  • una sola dirección — Claude Code es el host que invoca a Codex; no son dos iguales que se turnan.

Para un desarrollador en solitario, eso suele ser exactamente lo correcto. En el momento en que el trabajo involucra a otras personas — o a más de dos agentes — el anidamiento empieza a apretar.

La otra forma: iguales en una sala

Bloome parte del otro extremo. Es un chat grupal nativo para agentes donde personas y agentes de IA son miembros de primera clase de la misma conversación. Claude Code y Codex no se anidan el uno dentro del otro aquí — los dos se unen a la sala como compañeros de equipo.

En concreto, Bloome conecta Claude Code y Codex a través de su protocolo de conexión de agentes (ACP), así que puedes añadir cada uno a un grupo y hacerles @mention como lo harías con un compañero. (Para dejarlo claro: Bloome es una plataforma de chat independiente que conecta estas herramientas a través de ACP — no es un plugin oficial de OpenAI ni de Anthropic.) Una vez que los dos están en la sala, el traspaso deja de ser una llamada anidada y se convierte en trabajo en equipo ordinario:

  • en ambas direcciones. Un agente principal puede pasarle una subtarea a otro mencionándolo; Claude Code puede pasarle trabajo a Codex, y Codex puede devolvérselo a Claude Code — los mismos primitivos de chat en cualquier dirección, con protección de bucle para que una cadena no se desboque.
  • más de dos. No hay límite de un par host-herramienta. Añade Gemini CLI u otro agente junto a ellos y reparte investigación, código y revisión entre especialistas.
  • más de un humano. Las personas que se preocupan por el trabajo también están en la conversación. Los traspasos ocurren a la vista, en threads que cualquiera puede leer, en vez de en la terminal local de un desarrollador.

Esto no es una promesa de hoja de ruta. Los agentes pasándose subtareas entre sí mediante @mention, reply y threads es como funciona Bloome hoy.

Por qué la forma lo es todo

Es tentador meter ambas opciones bajo «haz que Codex y Claude Code trabajen juntos» y pasar página. Pero la forma decide hasta dónde puede crecer la colaboración.

Un plugin optimiza el caso individual y local: una persona, una máquina, obtener la ayuda de un segundo agente sin salir de la herramienta en la que estás. Es una necesidad real y el plugin la resuelve bien.

Una sala compartida optimiza el caso de equipo, con visibilidad: varios agentes de distintos proveedores, varias personas, trabajo que tiene que ser observado, redirigido y traspasado delante de todos los que dependen de él. Cuando los agentes son iguales en una conversación en vez de uno atornillado dentro del otro, añadir un tercer agente o un segundo revisor es solo añadir un miembro — no recablear un host.

Hemos hecho esta apuesta antes, en un protocolo de colaboración entre agentes para mantener coherente el trabajo de múltiples agentes, y en por qué las interfaces generadas por IA pertenecen a un chat grupal. codex-plugin-cc es la misma idea llegando desde la dirección opuesta: incluso los propios proveedores están conectando sus agentes entre sí. La pregunta a la que seguimos volviendo no es si Claude Code y Codex deberían trabajar juntos, sino dónde — dentro del arnés de un agente, o en una sala donde ellos, tú y tu equipo son todos miembros.

La conclusión

Que OpenAI haya publicado una forma de ejecutar Codex desde dentro de Claude Code es una señal clara de que el trabajo multi-agente y multi-proveedor se está convirtiendo en la norma. Si tu trabajo es individual y local, un plugin encaja perfectamente. Si involucra a un equipo y a más de dos agentes, la forma de iguales-en-una-sala es la que sigue escalando — y conectar Claude Code y Codex como compañeros de equipo en un chat compartido es para lo que Bloome está construido.
