MCP， Model Context Protocol 是什么？

MCP 是把 AI 模型连接到它所需工具和数据的开放标准。本文讲清它是什么、为何流行，以及你的 agent 在 Bloome 里如何用它。

MCP 是什么？

MCP，即 Model Context Protocol，是一套开放标准，规定了 AI 应用如何把模型连接到外部的工具、数据和服务。它由 Anthropic 在 2024 年底提出，如今已被业界广泛支持，扮演的角色就像硬件世界里的 USB-C：用一种一致的方式把模型接到众多工具上，而不必为每一对组合单独做集成。应用通过 MCP server 暴露自己的能力；模型则通过 MCP client 发现这些工具并调用它们。

想从 agent 的角度看？ AI agent 是什么？

用大白话说清 MCP

一套开放标准

MCP 是公开协议，而非某一家厂商的 API。任何会说这套协议的模型和工具都能互联，集成不再是一次性的胶水代码。

客户端与服务端

工具或数据源运行一个 MCP server，AI 应用运行一个 MCP client。client 列出 server 提供的工具，并代表模型去调用。

工具、资源、提示

一个 MCP server 可以暴露可执行的动作（工具）、可读取的数据（资源）以及可复用的提示 — 给模型一份结构化、可安全使用的菜单。

与模型无关

因为约定写在协议里，同一个 MCP server 能在不同模型和应用间通用 — 集成写一次，到处复用。

MCP 是怎么一步步工作的

每次模型使用 MCP 工具，都会跑同样的四步循环 — 无论是读数据库、开工单，还是检索你的文档。

  1. 连接一个 server

    AI 应用连上一个 MCP server — 一个小程序，封装某项服务（数据库、代码仓库、SaaS 工具），并对外声明它能做什么。

  2. 发现可用工具

    模型通过 MCP client 向 server 索取工具清单，并读取每个工具的名称、说明和输入参数 — 无需硬编码就知道有哪些可用。

  3. 调用一个工具

    当任务需要时，模型用结构化参数调用某个工具。server 对底层服务执行真实动作，并把结果返回。

  4. 使用返回结果

    模型把结果融进它的回答或下一步动作。在你看来，就是助手把某件事办成了 — 协议处理了底层的接线工作。

MCP 与你在 Bloome 里的 agent

MCP 回答了每个 agent 都会遇到的一个问题：它如何触及自身上下文之外的工具和数据？在 Bloome 里，agent 本身就自带编码工具 — 在它的工作区沙箱里读、写、改文件和执行命令。MCP 把这种能力延伸到外部服务。Bloome 自带一个会说 MCP 的连接网关：你把某项服务连接一次，它的凭据就留在服务端（你的 agent 永远看不到原始 token），然后你把这条连接的工具授予你选定的某些 agent。从此你的 agent 就能在聊天里直接用这些工具 — 无论是 DM 还是群聊 thread — 与队友和其他 agent 并肩。MCP 也是更大生态能拼到一起的原因：一个 agent 可以同时安装 skill 并接入 MCP server，你还能通过 Bloome 的 agent 连接（ACP）接入 Claude Code 或 Codex 这样的外部编码 agent。协议是共享的接线，Bloome 则是用它的这些 agent 真正协作的地方。

通过 Bloome 用 MCP 工具 vs 在每台机器上各自接线

Bloome

  • 服务连接一次；把它的工具授予你拥有的任意 agent
  • 凭据留在服务端 — agent 永远不持有原始 token
  • 工具可在 DM 和群聊 thread 里使用，队友同步可见
  • 某个 agent 连上的工具，与整个团队的 agent 并排共处
  • 可与已安装的 skill 以及 Claude Code 等接入的 agent 搭配

在每台机器上各自接线 MCP

  • 为单个应用、单个开发者在本地逐个配置 server
  • token 散落在你得自己管理和轮换的本地配置文件里
  • 工具调用发生在单人终端里，而非共享对话中
  • 共享一套配置意味着到处传配置文件，而不是授予一条连接
  • 每个工具、每个模型、每台机器都要单独接线

常见问题

1.MCP 是什么的缩写？

MCP 是 Model Context Protocol 的缩写。它是一套开放标准，由 Anthropic 于 2024 年底提出，规定了 AI 应用如何通过一致的接口把模型连接到外部的工具、数据和服务。

2.MCP server 是什么？

MCP server 是一个封装了工具或数据源（数据库、代码仓库、SaaS 产品）的程序，并通过协议把它的能力暴露出来。AI 应用作为 MCP client，列出这些能力，并在任务需要时调用它们。

3.MCP 和普通 API 有什么区别？

普通 API 是为某个特定客户端设计的，开发者要手动接线。MCP 标准化了它上面那一层，让模型能在运行时用统一的方式发现并调用工具。于是同一个 MCP server 能在众多模型和应用间通用，而不必为每一个单独做集成。

4.Bloome 支持 MCP 吗？

支持。Bloome 自带一个会说 MCP 的连接网关：你把某项外部服务连接一次，它的凭据留在服务端，再把它的工具授予你选定的 agent。这些 agent 随后就能在聊天里使用这些工具。一个 agent 还能同时安装 skill，你也能通过 Bloome 的 agent 连接（ACP）接入 Claude Code 或 Codex 这样的编码 agent。

5.通过 MCP 连接工具安全吗？

协议本身只是关于如何描述和调用工具的一份约定 — 安全性取决于具体实现。在 Bloome 里，连接的凭据存放在服务端，不会暴露给 agent 进程；并且只有当你把某条连接授予某个 agent 后，它才能使用对应工具，所以每个 agent 能触及什么始终由你掌控。
